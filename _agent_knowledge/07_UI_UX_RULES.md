# 07_UI_UX_RULES

**Назначение:** зафиксировать правила интерфейса и поведения UI, чтобы проект не превратился в перегруженный комбайн.  
**Когда читать:** при проектировании экранов, навигации, форм, состояний списков и визуальных улучшений.  
**Когда обновлять:** если меняются принципы интерфейса, обязательные ограничения по теме, приоритеты сценариев или правила представления данных.

## Базовые принципы интерфейса

- Первая версия не должна быть перегруженной.
- Главные бытовые сценарии должны выполняться быстро.
- Интерфейс должен помогать работать с постоянным списком покупок, а не заставлять пользователя каждый раз собирать всё заново.

## Правила по сценариям

- Создание новой позиции должно оставаться возможным из любого места.
- При этом создание новой карточки не должно быть акцентировано как основной сценарий.
- Основной сценарий — выбор и переиспользование уже существующих товаров из базы.

## Правила по представлениям

- Представления по магазинам, группам товаров, получателям и общему списку должны оставаться понятными.
- Интерфейс не должен превращаться в перегруженный комбайн из параллельных режимов и второстепенных действий.
- Нельзя ломать базовую бытовую логику приложения ради вторичных функций.
- Поведение срочных позиций в свёрнутых магазинных блоках не должно быть жёстко зашито: оно должно быть управляемым из UI-переключателя.
- В карточке магазина и связанных формах нужно использовать универсальную пользовательскую терминологию «Группы магазина» вместо «Отделы».

## Обязательные ограничения

- Нужно сохранять светлую тему.
- Не нужно предлагать тёмную тему в рамках текущей логики проекта.
- Будущие UI-решения должны проверяться на соответствие уже принятой логике приложения и ограничениям первой версии.
