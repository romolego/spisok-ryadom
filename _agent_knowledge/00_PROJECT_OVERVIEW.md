# 00_PROJECT_OVERVIEW

**Назначение:** кратко объяснить, что это за проект, зачем он создаётся и какой результат считается целевым.  
**Когда читать:** в самом начале работы, чтобы не предлагать решения вне согласованной концепции.  
**Когда обновлять:** если меняется базовое позиционирование проекта, целевая платформа, модель хранения данных, набор основных инструментов или общая формулировка цели.

## Что это за проект

«Список рядом» — локальное Android-приложение для ведения постоянного личного списка покупок. Это не одноразовый чеклист на один поход в магазин, а личная система управления покупками с собственной базой товаров, магазинами, отделами и рабочим текущим списком.

## Зачем создаётся проект

Проект нужен, чтобы пользователь мог:
- не переписывать список покупок заново;
- хранить собственную базу товаров и переиспользовать её;
- распределять покупки по магазинам;
- учитывать срочные и отложенные позиции;
- работать в бытовой логике реального похода по магазинам, а не только по алфавитному списку.

## Какой результат ожидается

Целевой результат — рабочее локальное Android-приложение, в котором данные хранятся на устройстве, основные бытовые сценарии выполняются быстро, а логика проекта остаётся управляемой и пригодной для публикации в RuStore после доведения до рабочего состояния.

## Контекст разработки

- Проект создаётся локально на ноутбуке.
- Основная IDE — Android Studio.
- Основная среда разработки и ведения задач — Claude Code.
- Codex, Cursor и другие ИИ-агенты могут использоваться как вспомогательные инструменты, если работают по этой же knowledge-базе.
- Пользователь не является программистом, поэтому агент берёт на себя практическую реализацию, аккуратное внесение изменений, поддержание структуры проекта и актуальности базы знаний.

## Базовая продуктовая логика

- В центре приложения находится один общий текущий список покупок.
- Вокруг него строятся карточки товаров, магазины, отделы магазинов, пользовательские группы товаров, получатели и локальные справочники.
- Данные хранятся локально.
- Первая рабочая версия должна оставаться управляемой по сложности и не расползаться в несвязанные функции.

## Принцип развития проекта

Проект ведётся строго итерационно:
- один запрос агенту — одно содержательное изменение;
- крупные несвязанные задачи не объединяются;
- сначала создаётся каркас, затем сущности, затем экраны, затем сценарии, затем улучшения интерфейса и вторичная логика.

Knowledge-инфраструктура проекта должна оставаться общей и одинаково понятной для разных ИИ-агентов, а не подстраиваться под один конкретный инструмент.
