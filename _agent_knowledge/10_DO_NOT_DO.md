# 10_DO_NOT_DO

**Назначение:** отдельно зафиксировать предметные запреты и ограничения проекта, чтобы агент не уводил разработку в неверную сторону.  
**Когда читать:** перед предложением новых функций, расширением модели, изменением архитектуры или интерфейса.  
**Когда обновлять:** если появляются новые подтверждённые запреты, меняются границы первой версии или снимаются ранее зафиксированные ограничения.

## Чего нельзя делать в первой версии

- Нельзя усложнять первую версию функциями, без которых не ломается базовый бытовой сценарий.
- Нельзя преждевременно тащить во вторую итерацию то, что не нужно для каркаса и основных сущностей.
- Нельзя превращать приложение в перегруженный функциональный комбайн.

## Чего нельзя делать с моделью приложения

- Нельзя ломать уже согласованную концепцию без явной фиксации нового решения.
- Нельзя плодить параллельные текущие списки в первой версии.
- Нельзя без необходимости вводить серверную часть.
- Нельзя сводить модель товара к одному магазину, если товар по концепции может быть связан с несколькими магазинами.
- Нельзя делать интернет обязательной частью базовой работы приложения.

## Чего нельзя делать с пользовательским сценарием

- Нельзя превращать создание новой карточки товара в основной путь взаимодействия.
- Нельзя жертвовать быстрыми бытовыми сценариями ради второстепенных режимов.
- Нельзя ломать понятные представления по магазинам, группам товаров, получателям и общему списку.
- Нельзя автоматически вычищать купленные позиции так, чтобы терялась понятная рабочая логика списка.

## Чего нельзя делать с процессом работы

- Нельзя смешивать крупные несвязанные изменения в одном запросе.
- Нельзя перескакивать через каркас сразу к вторичным улучшениям.
- Нельзя придумывать неподтверждённые факты вместо опоры на репозиторий и первоисточники.
- Нельзя привязывать project knowledge-базу только к одному агентскому инструменту.
- Нельзя дублировать один и тот же контекст в нескольких knowledge-файлах без практической причины.
