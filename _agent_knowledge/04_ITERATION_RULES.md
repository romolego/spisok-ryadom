# 04_ITERATION_RULES

**Назначение:** формализовать модель итеративной работы и не дать проекту расползаться по нескольким большим задачам сразу.  
**Когда читать:** при планировании любой новой итерации и перед формулировкой следующего шага.  
**Когда обновлять:** если меняется порядок развития проекта, критерии хорошей итерации или правила дробления задач.

## Базовая модель

- Один запрос агенту — одно содержательное изменение.
- Нельзя объединять в одну итерацию несколько крупных несвязанных задач.
- Любое ТЗ для агента должно относиться к одной текущей итерации.
- Каждая итерация должна быть изолированной, проверяемой и понятной по результату.

## Порядок развития проекта

Проект должен расти в такой последовательности:
1. каркас;
2. сущности;
3. экраны;
4. сценарии;
5. улучшения интерфейса и вторичная логика.

Если задача перескакивает через этот порядок без явной причины, её нужно пересобрать в более узкий шаг.

## Как определять границы итерации

Хорошая итерация:
- даёт один конкретный результат;
- не ломает ранее согласованную концепцию;
- не требует в той же задаче параллельно решать архитектуру, UI и вторичные функции;
- позволяет проверить результат сразу после завершения.

## Что делать со слишком широкой задачей

- Если задача слишком широкая, её нужно дробить на этапы.
- Агент не должен предлагать большой комбайн вместо узкой доработки.
- Если следующий логичный шаг ещё не готов, нужно сначала закрыть текущий опорный слой, а не тянуть будущие подсистемы в ту же итерацию.
