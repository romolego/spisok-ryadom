# 08_DECISION_LOG

**Назначение:** вести журнал ключевых проектных решений с объяснением причин и влияния на дальнейшую разработку.  
**Когда читать:** если задача затрагивает продуктовые ограничения, спорные места модели, приоритеты первой версии или уже согласованные компромиссы.  
**Когда обновлять:** когда принимается новое значимое решение или меняется ранее зафиксированное решение.

## D-001. Один общий текущий список в первой версии

- **Что решено:** в первой версии используется один общий текущий список покупок.
- **Почему:** это упрощает модель и позволяет закрыть потребности через группировки, фильтры и представления, а не через множество параллельных списков.
- **Влияние:** агент не должен вводить несколько текущих списков без отдельного нового решения.

## D-002. Все данные хранятся локально

- **Что решено:** данные приложения хранятся локально на устройстве.
- **Почему:** это часть базовой концепции продукта и требования к работе без интернета.
- **Влияние:** первая версия не должна требовать серверную часть как обязательную основу.

## D-003. Целевая публикация — RuStore

- **Что решено:** проект делается как Android-приложение с расчётом на публикацию в RuStore.
- **Почему:** это целевая модель выпуска, зафиксированная в концепции.
- **Влияние:** приложение должно оставаться понятным, стабильным и без лишних разрешений.

## D-004. Проект ведётся строго итерациями

- **Что решено:** работа идёт через узкие изолированные итерации.
- **Почему:** пользователь не программист, а проект нужно удерживать управляемым и проверяемым.
- **Влияние:** агент не должен смешивать несколько крупных несвязанных задач в одном изменении.

## D-005. Пользовательские классификаторы задаёт сам пользователь

- **Что решено:** получатели, группы товаров, единицы измерения, названия магазинов и названия отделов не должны быть жёстко зашиты как пользовательские значения.
- **Почему:** приложение должно быть гибким и пригодным для широкой аудитории.
- **Влияние:** агент не должен вводить фиксированные пользовательские списки там, где концепция требует пользовательского ввода.

## D-006. Приоритетный магазин используется как магазин по умолчанию

- **Что решено:** если у товара задан приоритетный магазин, при добавлении из общего контекста он подставляется как магазин по умолчанию.
- **Почему:** это ускоряет главный бытовой сценарий и поддерживает работу с товарами, которые доступны в нескольких магазинах.
- **Влияние:** связь товара с магазином должна хранить приоритет, а не сводиться к одному полю магазина в карточке товара.

## D-007. Новые позиции можно создавать из любого места, но это не основной сценарий

- **Что решено:** новая карточка товара может создаваться из разных представлений, но основной сценарий всё равно строится вокруг выбора из уже существующей базы.
- **Почему:** так сохраняется скорость бытового использования и снижается количество дублей.
- **Влияние:** UI не должен продвигать постоянное создание новых карточек как главный путь взаимодействия.

## D-008. Основная среда разработки — Claude Code

- **Что решено:** основной средой агентской разработки считается Claude Code.
- **Почему:** это часть согласованного рабочего процесса проекта.
- **Влияние:** документация должна явно учитывать этот контекст, не исключая при этом другие агенты.

## D-009. Codex и Cursor используются как вспомогательные агенты

- **Что решено:** Codex и Cursor используются как вспомогательные ИИ-инструменты.
- **Почему:** проект допускает работу несколькими агентами, но на общей knowledge-базе.
- **Влияние:** любой агент обязан ориентироваться на одну и ту же агентскую инфраструктуру внутри репозитория.

## D-010. База знаний должна оставаться tool-neutral

- **Что решено:** проектная knowledge-инфраструктура не должна зависеть от одного конкретного агентского инструмента.
- **Почему:** над проектом могут работать Claude Code, Codex, Cursor и другие агенты, и все они должны видеть один и тот же контекст.
- **Влияние:** нельзя писать правила, которые скрыто завязаны только на один агентский интерфейс или одну терминологию.

## D-011. Магазины могут быть как с отделами, так и без них

- **Что решено:** у магазина может быть набор отделов, но наличие отделов не является обязательным.
- **Почему:** в реальной жизни часть торговых точек имеет внутреннюю структуру, а часть нет.
- **Влияние:** модель и UI должны поддерживать оба варианта без усложнения базового сценария.

## D-012. Купленные позиции не удаляются автоматически сразу

- **Что решено:** купленная позиция по умолчанию не исчезает моментально из текущего списка.
- **Почему:** это помогает в реальном походе по магазину и снижает риск ошибочного исчезновения нужной позиции.
- **Влияние:** интерфейс должен поддерживать состояние купленной позиции и отдельную очистку купленных записей.

## D-013. Технологический стек первой версии — Kotlin + Compose + Room

- **Что решено:** первая версия реализована на Kotlin с Jetpack Compose (Material 3), Room (SQLite), Compose Navigation и MVVM (ViewModel + StateFlow).
- **Почему:** Compose — современный стандарт для новых Android-проектов, хорошо интегрируется с ViewModel, Navigation и Room, требует меньше boilerplate для списков и форм. Проект начинается с нуля, нет legacy XML.
- **Влияние:** все экраны реализуются как Composable-функции, данные передаются через StateFlow, навигация через Compose Navigation.

## D-014. Manual DI (AppContainer) вместо Hilt

- **Что решено:** DI реализован через ручной AppContainer в Application subclass, без Hilt/Dagger.
- **Почему:** в приложении одна база данных, 4 репозитория и 5 ViewModel — ручной контейнер проще и не требует annotation processing через kapt/ksp для DI. При росте приложения можно мигрировать на Hilt.
- **Влияние:** ViewModels получают зависимости через ViewModelProvider.Factory, обращаясь к AppContainer.

## D-015. purchaseType хранится как String

- **Что решено:** тип покупки (онлайн/оффлайн) хранится в Room как String ("offline"/"online"), а не как Room enum.
- **Почему:** forward-compatibility — при добавлении новых типов не потребуется миграция базы.
- **Влияние:** UI использует radio-кнопки для выбора, бизнес-логика сравнивает строки.

## D-016. Фото товара — URI-строка без интеграции с камерой/галереей в v1

- **Что решено:** поле `photoUri` заложено в модели ProductEntity, но в первой итерации UI показывает текстовое поле для ввода URI, без камеры/галереи.
- **Почему:** интеграция с камерой и галереей усложняет первую итерацию без критической необходимости.
- **Влияние:** камера/галерея будут добавлены в следующих итерациях.

## D-017. История отложена из каркаса первой итерации

- **Что решено:** сущности истории (HistorySnapshot, HistoryEntry) не включены в Room-базу первой итерации.
- **Почему:** каркас первой итерации фокусируется на рабочих сущностях (товары, магазины, текущий список). История — задел, который можно добавить позже без ломки основной модели.
- **Влияние:** при добавлении истории потребуется Room-миграция (добавление новых таблиц).

## D-018. Текущий список группируется по магазинам как основное представление

- **Что решено:** главный экран текущего списка показывает позиции, сгруппированные по магазинам (по displayOrder), с группой «Без магазина» в конце.
- **Почему:** представление по магазинам — главный рабочий сценарий приложения (пройтись по магазинам и отмечать купленное).
- **Влияние:** сортировка внутри группы: срочные некупленные → обычные некупленные → купленные.

## D-019. Демо-данные через SharedPreferences с лёгким отключением

- **Что решено:** демо-данные заполняются при первом запуске через DemoDataProvider с флагом в SharedPreferences. Повторная загрузка не происходит.
- **Почему:** нужно показать приложение как рабочий продукт с адекватной тестовой картиной. Механизм легко отключить комментированием одной строки или вызовом resetDemoFlag.
- **Влияние:** при необходимости сброса данных: вызвать DemoDataProvider.resetDemoFlag() и пересоздать БД.

## D-020. Единицы измерения — подсказки вместо жёсткого справочника

- **Что решено:** при вводе единицы измерения показываются уже существующие значения из базы как подсказки. Пользователь может ввести любое новое значение.
- **Почему:** концепция требует модели рекомендаций, а не жёсткого закрытого справочника.
- **Влияние:** компонент QuantityUnitInputWithSuggestions используется в карточке товара и в диалоге добавления в список.

## D-021. Быстрый перенос между магазинами с автоподстановкой отдела

- **Что решено:** при переносе позиции в другой магазин сначала проверяются product-shop links для отдела, затем ищется одноимённый отдел в новом магазине.
- **Почему:** ускоряет сценарий «не нашёл товар — переношу в другой магазин» с минимумом действий.
- **Влияние:** допустимые магазины (из product-shop links) показываются первыми в диалоге переноса.

## D-023. Количество и единица в записи списка независимы от карточки товара

- **Что решено:** поля quantity и unit в ShoppingListEntryEntity — это фактические значения конкретной покупки, независимые от defaultQuantity/defaultUnit в ProductEntity. Редактирование этих полей у записи списка не меняет карточку товара.
- **Почему:** пользователь часто покупает товар в разном количестве, и изменение количества для одной покупки не должно влиять на дефолтные настройки карточки.
- **Влияние:** при добавлении товара в список дефолтные значения используются как стартовая подстановка. Дальше пользователь редактирует количество прямо из списка через диалог, без необходимости заходить в карточку товара.

## D-024. Основное действие записи списка — работа с текущей покупкой, а не с карточкой товара

- **Что решено:** короткий тап по любой основной зоне карточки записи текущего списка открывает компактный диалог редактирования записи (количество, магазин, заметка). Карточка товара и удаление доступны через long press контекстное меню. Чекбокс «куплено» и иконка срочности работают как отдельные элементы и не конфликтуют с общим тапом по карточке.
- **Почему:** в основном бытовом сценарии пользователю нужно быстро скорректировать количество и отметить купленное, а не редактировать карточку товара. Переход к карточке и удаление — вторичные действия. Зона клика по всей карточке делает быстрое редактирование основным и удобным действием.
- **Влияние:** ShoppingListItem использует combinedClickable на Card с onClick = onEditEntry и onLongClick для контекстного меню. Бейдж количества справа остаётся как визуальный индикатор, но не имеет собственного обработчика клика. Удаление убрано из основного UI (был крестик справа). В быстром диалоге единица измерения показывается как контекст, а не как редактируемое поле.

## D-025. Перестановка и сворачивание магазинных блоков в текущем списке

- **Что решено:** в режиме «по магазинам» пользователь может менять порядок магазинных блоков кнопками вверх/вниз и сворачивать/разворачивать блоки кликом по заголовку. Порядок сохраняется через displayOrder в Room, состояние сворачивания — в SharedPreferences.
- **Почему:** основной рабочий сценарий — прохождение по магазинам в удобном порядке. Перестановка позволяет настроить маршрут покупок. Сворачивание уменьшает визуальный шум, когда часть магазинов уже пройдена или не актуальна.
- **Влияние:** ShopDao получил метод updateAll для пакетного обновления порядка. EntryGroup расширен полями isCollapsed, totalCount, boughtCount. Блок «Без магазина» всегда последний и не перемещается. Функциональность ограничена режимом «по магазинам».

## D-022. Четыре переключаемых представления текущего списка

- **Что решено:** текущий список поддерживает 4 режима представления одних и тех же данных: по магазинам, по группам товаров, по получателям, общий без группировки. Режим переключается через FilterChip-панель и сохраняется в SharedPreferences.
- **Почему:** концепция требует нескольких представлений (06_FUNCTIONAL_REQUIREMENTS), но как разные виды одного общего списка, а не как отдельные списки. Представление по магазинам остаётся главным.
- **Влияние:** модель ShopGroup обобщена до EntryGroup; ShoppingListEntryUi расширен полями groupName и recipientName; ViewModel получил зависимость от ClassifierRepository; очистка купленных по группе работает только в режиме «по магазинам».
